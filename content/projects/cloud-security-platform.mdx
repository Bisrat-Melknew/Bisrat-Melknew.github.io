## Overview

Designed and deployed a unified Security Operations Center (SOC) platform that provides centralized visibility, automated threat detection, and coordinated incident response across AWS and Azure cloud environments supporting 200+ microservices.

## Problem

The organization operated workloads across both AWS and Azure with siloed security tooling:

- **Fragmented visibility** — Security teams used separate consoles for each cloud provider
- **Alert fatigue** — Over 10,000 daily alerts with no correlation or prioritization
- **Manual response** — Incident response procedures were documented but not automated
- **Compliance gaps** — No unified view of compliance posture across environments
- **Slow detection** — Average time to detect threats was 4+ hours

## Solution

Built a multi-layered security operations platform:

1. **Unified SIEM** — Centralized log aggregation from both clouds into Splunk with custom correlation rules
2. **Automated Response** — Python-based SOAR playbooks for common incident types
3. **Cloud Security Posture** — Continuous compliance scanning against CIS benchmarks
4. **Threat Intelligence** — Automated IOC enrichment and threat hunting workflows

## Technologies Used

- **AWS**: CloudTrail, GuardDuty, Config, Security Hub, WAF, KMS
- **Azure**: Sentinel, Defender for Cloud, Monitor, Key Vault
- **SIEM**: Splunk Enterprise Security with custom apps
- **Automation**: Python (boto3, azure-sdk), AWS Lambda, Azure Functions
- **IaC**: Terraform for all infrastructure, Helm for Kubernetes workloads
- **Container Security**: Trivy, Falco runtime monitoring

## Security Considerations

- All cross-cloud data transfer encrypted in transit via private endpoints
- Least-privilege IAM roles with automated access reviews
- Secrets managed through HashiCorp Vault with dynamic credentials
- Multi-factor authentication enforced for all SOC access
- Audit logging for all platform administrative actions
- Data retention policies aligned with regulatory requirements

## Challenges

1. **Log format normalization** — Standardizing event formats across two cloud providers required extensive parsing logic
2. **False positive reduction** — Initial deployment generated excessive alerts; iterative tuning reduced noise by 85%
3. **Cross-cloud correlation** — Linking related events across AWS and Azure required custom identity mapping

## Results

- **Mean Time to Detect** reduced from 4 hours to 12 minutes
- **Mean Time to Respond** reduced from 2 hours to 8 minutes for automated playbooks
- **85% reduction** in false positive alerts
- **100% compliance** score across CIS benchmarks for both cloud environments
- **Zero missed critical incidents** in first 12 months of operation

## Lessons Learned

- Invest heavily in log normalization before building detection rules — garbage in, garbage out
- Start with high-fidelity detection rules and expand gradually rather than enabling everything at once
- Automated response should always include human-in-the-loop for destructive actions
- Regular purple team exercises validate detection coverage and response procedures
